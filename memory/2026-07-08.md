## Doubao 复刻音色 Resource 匹配纠偏

### 提问和更改方向

用户在豆包 adapter 合成时遇到 `Doubao SSE event returned error code 55000000: resource ID is mismatched with speaker related resource.`。这说明上一轮把 `seed-icl-*` 简化为“只能复刻，不能合成”的判断不完整；豆包复刻音色后续合成时仍需要使用对应的 `seed-icl-*` 作为 `X-Api-Resource-Id`。

### 当前项目状况和进展

- 豆包 `seed-tts-*` 资源用于普通 TTS 音色合成。
- 豆包 `seed-icl-*` 资源用于声音复刻，也用于该系列复刻音色的后续 SSE 合成。
- `X-Api-Resource-Id` 必须和 `speaker` 所属资源匹配，否则会出现用户反馈的 55000000 错误。

### 当前步骤所做内容

- 恢复 `seed-icl-*` 在 capability 中对 `tts.sync`、`tts.stream` 的支持，同时保留 `voice.clone.create` 支持。
- `seed-icl-*` 不作为普通 TTS 默认模型，但会以“复刻音色资源”显示。
- 豆包复刻生成的 `VoiceRecord.modelId` 改回实际复刻资源 `seed-icl-*`，不再映射成 `seed-tts-*`。
- 豆包 TTS vendor extension 的 `resourceId` 允许 `seed-icl-*`，用于复刻音色合成。
- `TTSFacade.resolveVoiceSelection()` 在使用本地 `voiceId` 时，会把请求 model 自动切换为该 voice 绑定的 `modelId`。
- 快速合成页和 Benchmark 配置页在选择本地受控音色后，会自动切换到该音色绑定的合成 Resource Id。
- 更新豆包 adapter 测试，覆盖 `seed-icl-2.0` 可用于同资源复刻 speaker 的 TTS 合成。
- 更新豆包 Contract，明确 speaker/resource 必须匹配。

### 缺陷和风险点

- 对只输入原始 `providerVoiceId`、不选择本地受控 `voiceId` 的请求，平台无法知道该 speaker 属于哪个豆包 Resource Id，用户仍需手动选择正确 `model`。
- 已经被上一轮代码错误写成 `seed-tts-*` 的本地历史 voice registry 记录，需要手动或迁移脚本修正回对应 `seed-icl-*`。

### 下一步内容或计划内容

- 可以在 Doubao adapter 中针对 55000000 错误补充更友好的错误说明，提示检查 speaker 与 `X-Api-Resource-Id` 是否匹配。
- 若后续做 voice registry 迁移，可以根据 `vendorMetadata.cloneResourceId` 或历史来源修复 `modelId`。

### 推荐内容

- 豆包 UI 上应把左侧“模型”标注为 Resource Id，避免和 `req_params.model` 混淆。
- 复刻音色合成优先选择本地 voice registry 中的 `voiceId`，不要手输 providerVoiceId；这样平台可以自动带上正确 Resource Id。

## 厂商级音色语义纠偏

### 提问和更改方向

用户进一步澄清现行产品逻辑：音色归属于厂商，而不是归属于复刻模型。复刻时可以选择复刻模型创建音色；复刻完成后的音色，可以在同一厂商范围内任意 TTS 合成模型中使用。因此上一节把豆包复刻音色和 `seed-icl-*` 合成 Resource Id 绑定的实现方向需要撤销。

### 当前项目状况和进展

- 平台应把 `voiceId` 视为 provider 级受控音色引用。
- 合成表单的模型候选只由 `tts.sync` / `tts.stream` capability 决定，不由音色历史 `modelId` 反向决定。
- 复刻表单的模型候选只由 `voice.clone.create` capability 决定。

### 当前步骤所做内容

- `TTSFacade.resolveVoiceSelection()` 改为只把本地 `voiceId` 解析为 `providerVoiceId`，不再覆盖请求中的 `model`。
- 豆包 `seed-icl-*` capability 恢复为只支持 `voice.clone.create`；`seed-tts-*` 继续作为 TTS 合成模型。
- 豆包复刻结果不再写 `VoiceRecord.modelId`，只在 `vendorMetadata.cloneModelId` 记录复刻模型用于审计。
- 快速合成页和 Benchmark 配置页移除“选择本地音色后自动切换模型”的逻辑；本地音色下拉只按 provider 加载。
- 音色管理页的合成测试改用厂商默认 TTS 模型，不再使用受控音色上的历史 `modelId`。
- 更新相关 Vitest：覆盖本地 voice 不改写模型、`seed-icl-*` 不能作为 TTS 模型、TTS vendor resource 不能覆盖成 clone 模型、provider 级音色选项。

### 缺陷和风险点

- 如果豆包真实接口仍返回 speaker/resource mismatch，需要后续在 adapter 内做符合平台语义的厂商侧映射或错误诊断，不能通过 UI 把音色绑定到复刻模型来解决。
- 历史本地 voice registry 中可能仍有 `modelId: seed-icl-*` 或 `vendorMetadata.cloneResourceId`，当前实现会忽略这些字段参与合成模型选择。

### 下一步内容或计划内容

- 如需完全兼容历史 registry，可增加一次性迁移，把 `vendorMetadata.cloneResourceId` 改名或补录为 `cloneModelId`，并移除 clone voice 上的 `modelId`。
- Benchmark 配置创建 API 后续仍应增加服务端 capability 校验，防止绕过 UI 提交 clone-only 模型作为 TTS 合成配置。

### 推荐内容

- 后续所有 UI 和 API 都应保持“音色 provider 级、模型 operation 级”的边界。
- 复刻模型只影响创建音色的请求和审计记录，不应影响后续合成模型选择。

## Voice Compatibility 能力模型落地

### 提问和更改方向

用户确认应在 core 能力模型中扩展，由 vendor adapter 声明并实现具体的音色绑定或匹配规则。最终方向不是全局假设音色绑定模型，也不是全局假设 provider 级通用，而是把厂商差异表达成 `VoiceCompatibility`。

### 当前项目状况和进展

- Doubao 真实失败来自 `speaker` 与 `X-Api-Resource-Id` 不匹配，属于 resource 绑定。
- CosyVoice 真实失败来自 flash voice 跑 plus model，属于 target_model 同模型绑定。
- MiniMax 文档未确认强绑定，只能作为推荐模型处理。
- Xiaomi MiMo 的 voiceclone 是即时复刻合成，不产生持久 voiceId。

### 当前步骤所做内容

- 在 `packages/core` 新增 `VoiceCompatibility` 和 `VoiceCompatibilityPolicy`，并扩展到 `VoiceRecord`、`TTSVoiceSelection`、`VoiceCloneCapability`、`TTSVendorModel`、`TTSOperationCapability`、`TTSCapabilities`。
- Adapter contract 新增可选 `voiceCompatibility(voice)`，由厂商 adapter 推导历史 voice registry 的兼容事实。
- `TTSFacade.resolveVoiceSelection()` 在解析本地 `voiceId` 时注入 `voice.compatibility`，并对 `scope=model` / `scope=resource` 做后端模型校验和默认补齐。
- Doubao adapter 对复刻音色写入 `scope=resource`，合成 plan 时优先把 `voice.compatibility.resourceIds[0]` 映射到 `X-Api-Resource-Id`；`request.model` 仍表示平台 TTS 模型候选。
- Doubao adapter 可从历史 `providerVoiceId` / `displayName` / metadata 推导 `seed-icl-*` 资源，修复旧 registry 无 compatibility 的情况。
- CosyVoice adapter 对复刻音色写入 `scope=model`，并在 plan 阶段拒绝跨 target_model 合成。
- MiniMax adapter 对 `modelId` 只写 provider 级 preferred model，不做强拦截。
- 前端 `modelOptions()` / `defaultModelForOperation()` 支持传入当前选中 voice，并只在强 compatibility 下过滤模型。
- 快速合成、Benchmark 配置、音色测试按选中 voice 的 compatibility 选择/过滤模型，修复 CosyVoice flash voice 被默认拿去跑 plus 的问题。

### 缺陷和风险点

- Doubao 的 `request.model` 与 `X-Api-Resource-Id` 仍存在历史语义混用；本次通过 `voice.compatibility` 优先映射 resource，后续可进一步把 canonical model 与 vendor resource 拆成更清晰的字段。
- 对只手输 `providerVoiceId` 而不选择本地 `voiceId` 的 Doubao 请求，平台无法自动推导 resource，只能依赖用户在 vendor extension 中填写 `resourceId`。
- MiniMax 跨模型是否完全可用仍需真实回归，当前按“不强制绑定”处理。

### 下一步内容或计划内容

- 可以给音色管理页增加 compatibility 展示列，明确“模型绑定 / 资源绑定 / provider 级 / unknown”。
- Benchmark 配置创建 API 后续应校验 `BenchConfig.voice` 与 `modelId` 的 compatibility，防止保存不可执行配置。
- 可考虑一次性迁移 `data/voices/voices.json`，把已知历史音色补写 `compatibility`，减少运行时推导。

### 推荐内容

- 后续接新厂商时必须先回答：clone result 是 provider-wide、same_model、same_resource、instant_only 还是 unknown。
- UI 过滤只能作为体验优化；adapter plan 和 facade compatibility 校验必须作为最终防线。

## Voice Compatibility Operation 级声明补齐

### 提问和更改方向

用户确认按照 core 能力模型扩展继续实现。补充方向是把 voice compatibility 不只放在 provider 或 clone result 上，也显式落在 operation/model capability 层，符合“能力声明必须按 operation 组织”的仓库约束。

### 当前项目状况和进展

- `VoiceCompatibilityPolicy` 已在 core 类型、adapter contract、facade、UI helper 中贯通。
- CosyVoice、Doubao、MiniMax、Xiaomi MiMo 已分别形成 same_model、same_resource、unknown、instant_only 四种策略。
- 前端合成和 Benchmark 表单已按 operation 的 `supportedOperations` 过滤模型，并按强 compatibility 进一步收窄候选。

### 当前步骤所做内容

- CosyVoice capability 在 provider、model、`tts.sync`、`tts.stream` 和 clone result 层复用 same_model 策略。
- Doubao capability 在 provider、TTS model、`tts.sync`、`tts.stream` 和 clone result 层复用 same_resource 策略，同时继续保证 `seed-icl-*` 只支持 `voice.clone.create`，不进入普通 TTS 模型候选。
- MiniMax capability 明确声明 unknown/none，不强制模型绑定，只保留推荐模型语义。
- Xiaomi MiMo capability 明确声明 instant_only，并把 `voice.clone.instant` 的结果标记为不可复用持久 voiceId。
- 增加四个 adapter capability 测试，分别覆盖 same_model、same_resource、unknown、instant_only。

### 缺陷和风险点

- Xiaomi MiMo provider 级 `instant_only` 表达的是 voiceclone 结果生命周期，不代表普通 TTS 音色需要绑定；后续 UI 展示时需要按 operation 解读。
- Doubao `seed-tts-*` 平台 TTS 模型与 vendor `req_params.model=seed-tts-2.0-standard/expressive` 仍需要在文案上持续区分。

### 下一步内容或计划内容

- 可以在 capability 查看或音色管理页增加 compatibility policy 展示，避免用户把 clone model、TTS Resource Id、厂商子模型混为一谈。
- Benchmark plan 创建时应继续加入服务端校验，确保保存前就能发现 voice/model/resource 不兼容。

### 推荐内容

- 新 vendor adapter 的 capability 测试应固定断言 operation 级 voice compatibility policy。
- 对 instant clone provider，不要生成持久 voice registry 记录；只归档即时音频和 mapping report。

## Doubao 历史音色 ID 规范化

### 提问和更改方向

用户反馈豆包仍报 `Doubao SSE event returned error code 55000000: resource ID is mismatched with speaker related resource`。需要基于最新 run archive 排查真实请求，而不是继续假设只是 `X-Api-Resource-Id` 未设置。

### 当前项目状况和进展

- 最新失败 run `run_20260708121711_8045f2e7` 已经把 `X-Api-Resource-Id` 设置为 `seed-icl-2.0`。
- 失败请求中的 `req_params.speaker` 为 `doubao_seed-icl-2.0_S_D3lMr9g32`，这是平台历史登记/展示 ID，不一定是豆包真实 speaker id。
- 按豆包文档，合成请求中的 `req_params.speaker` 应传训练得到的 `speaker_id` 或自定义音色 ID；对预付费槽位应是 `S_...`。

### 当前步骤所做内容

- 在 Doubao adapter 的 `normalizeProviderVoiceId()` 中增加历史格式兼容：当音色 ID 匹配 `doubao_<ResourceId>_<speaker>` 时，发给厂商的 speaker 改为后缀真实 `<speaker>`。
- 该兼容仅对 Doubao 已知 Resource Id 生效，包括 `seed-tts-*` 和 `seed-icl-*`。
- 增加 Doubao adapter 测试，覆盖 `doubao_seed-icl-2.0_S_D3lMr9g32` 会被规范化为 `S_D3lMr9g32`，同时 `X-Api-Resource-Id` 仍为 `seed-icl-2.0`。

### 缺陷和风险点

- 如果该账号下真实 speaker 并不是 `S_D3lMr9g32`，仍会失败；但当前 archive 显示原值明显包含平台拼接前缀，先剥离是正确方向。
- 手动登记外部豆包音色时，用户最好直接填豆包真实 speaker id，而不是填平台展示名。

### 下一步内容或计划内容

- 可以在音色管理页对 Doubao 外部登记增加提示：`providerVoiceId` 应填写豆包真实 `speaker_id` / `custom_speaker_id`。
- 如果后续仍失败，应调用或补接豆包 `get_voice` 查询接口，验证 speaker 实际绑定的 resource 和状态。

### 推荐内容

- Archive 排查豆包问题时优先看三项：`headers.X-Api-Resource-Id`、`body.req_params.speaker`、`body.req_params.model`。
- 不要把本地展示 ID 当成厂商 speaker id 写入请求体。
