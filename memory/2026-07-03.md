## 修复合成页使用已克隆音色 ID 的问题

### 提问和更改方向

用户反馈语音合成页面中，“厂商音色 ID”填写自己已经克隆好的音色后会出现问题。排查方向是合成页、voice registry、facade 和 MiniMax adapter 的音色 ID 映射链路。

### 当前项目状况和进展

- 音色克隆成功后，后端会把 `VoiceRecord` 保存到 `data/voices/voices.json`。
- 旧合成页只提供手填厂商音色 ID，没有列出本地已克隆音色。
- 旧 facade 在 `tts.sync` 前不会解析本地 `voiceId`，如果用户复制 `minimax:xxx` 这类平台 voice id 给 MiniMax，就会把错误 ID 原样发给厂商。

### 当前步骤所做内容

- `InMemoryVoiceRegistry` 增加 `get(voiceId)`，支持 facade 按平台 voice id 查本地记录。
- `TTSFacade.synthesizeSync()` 在 adapter plan 前解析 `request.voice.voiceId`，命中本地 registry 时改写为对应 `providerVoiceId`。
- MiniMax adapter 增加合成音色 ID 归一化，误填 `minimax:xxx` 时会剥离 provider 前缀后再写入 `voice_setting.voice_id`。
- Web `voices.ts` 增加 `listVoices()`，合成页会按当前 provider 加载已克隆音色。
- 合成页把音色输入从纯文本改为可输入的 combobox：选择本地已克隆音色时提交 `voice.voiceId`，手动输入时继续提交 `voice.providerVoiceId`。
- 新增/更新测试覆盖 voice registry get、facade 本地 voice id 解析、MiniMax 前缀剥离、前端音色选项生成。
- 运行 `pnpm typecheck` 和 `pnpm test`，均通过。

### 缺陷和风险点

- 合成页仍没有内置 MiniMax 系统音色完整列表，只展示本地已克隆音色，并允许用户手动输入系统或厂商音色 ID。
- MiniMax 克隆音色创建后，厂商侧可能存在短暂可用性延迟；如果刚克隆立刻合成仍失败，需要用真实接口继续观察错误码。
- 如果其他厂商真实 providerVoiceId 本身允许包含 `providerId:` 前缀，当前 MiniMax-only 剥离逻辑不会影响其他 adapter。

### 下一步内容或计划内容

- 用真实克隆结果在合成页选择本地音色候选，完成一次 MiniMax 端到端验收。
- 后续可在 provider capabilities 或单独 voice catalog 中加入常用系统音色，和本地 cloned voices 一起展示。

### 推荐内容

- 已克隆音色优先从合成页下拉候选中选择，避免手动复制平台 voice id 和厂商 providerVoiceId 时混淆。
- 若手动输入，MiniMax 可以输入纯厂商音色 ID；误输入 `minimax:xxx` 时当前 adapter 会兜底转成 `xxx`。

## 修复运行记录列表 Internal Error

### 提问和更改方向

用户反馈“运行记录 internal error”。排查方向是 `/v1/runs` 列表接口和 Web 运行记录页面。

### 当前项目状况和进展

- `data/runs` 现在既会保存同步 TTS 运行，也会保存音色克隆运行。
- 旧 `listRuns()` 把所有 `result.json` 都当作 `TTSSyncResult`，默认存在顶层 `createdAt` 和 `audio`。
- 音色克隆 archive 没有音频文件，历史克隆结果也可能没有顶层运行元数据，导致列表排序或前端显示时触发 internal error。

### 当前步骤所做内容

- 在 `packages/core` 新增 `ArchivedRunSummary`，作为运行列表的跨 operation 摘要类型。
- `FileRunArchive.writeVoiceClone()` 写入 `runId`、`providerId`、`operation`、`status`、`createdAt` 等顶层运行元数据。
- `FileRunArchive.listRuns()` 改为读取 `plan.json` 兜底，兼容历史 clone archive 中缺少 `createdAt/audio` 的情况。
- Web `RunsPage` 增加 operation 类型列，同步合成显示音频信息，音色克隆显示“无音频文件”。
- Web `RunDetailPage` 对 `audio` 做可选判断，避免非音频运行详情访问 `audio.url`。
- 新增测试覆盖：
  - 新版 voice clone archive 不需要音频也能出现在列表。
  - 历史 clone archive 缺少 run metadata 时，列表能从 plan metadata 兜底。
- 运行 `pnpm typecheck`、`pnpm test`、`pnpm --filter @tts-platform/web build`，均通过。

### 缺陷和风险点

- 运行详情页目前仍以 JSON viewer 为主，非 TTS 运行不会展示专门的音色克隆摘要卡片。
- 旧 archive 如果缺少 `plan.json`，列表仍会跳过该 run；这是合理兜底，因为 plan 是 archive contract 的核心文件。
- Web build 仍有 Vuetify/MDI chunk 大小提示，不影响功能。

### 下一步内容或计划内容

- 后续可给运行详情页增加 operation-aware 摘要：同步合成展示音频播放器，音色克隆展示 voice id、providerVoiceId、参考音频 id 和 clone workflow。
- 若后续接入 stream archive，可继续复用 `ArchivedRunSummary`，并按 operation 扩展列表显示。

### 推荐内容

- Archive 列表不要再绑定单一 `tts.sync` 结果结构，所有新 operation 都应写入统一顶层运行元数据。
- 前端运行记录只展示跨 operation 摘要，具体审计信息继续从详情页 JSON viewer 查看。

## 合成页下沉语音合成记录并修复音色选择崩溃

### 提问和更改方向

用户要求把语音合成记录放到语音合成页面下方，并反馈选择克隆音色后点击运行会报错：`(voiceIdInput.value ?? "").trim is not a function`。

### 当前项目状况和进展

- 合成页已经支持从本地 voice registry 加载已克隆音色候选。
- Vuetify `v-combobox` 选中对象项时，`v-model` 可能是 `{ title, value }`，不是字符串。
- 原提交逻辑直接对 `voiceIdInput.value` 调用 `.trim()`，因此选择下拉项时会出现运行时错误。

### 当前步骤所做内容

- 新增 `synthesize-submit.ts`，用 `voiceInputValue()` 统一处理手输字符串、下拉对象和清空 null。
- 合成页 `v-combobox` 增加 `item-title` 和 `item-value`，提交时通过 helper 提取实际音色 id。
- 新增 `synthesize-runs.ts`，筛选 `tts.sync` 记录并格式化音频列。
- 合成页底部新增“语音合成记录”表格，展示运行 ID、厂商、创建时间、音频信息和详情入口。
- 左侧导航去掉独立“运行记录”入口；详情路由仍保留，表格中的打开按钮继续进入 `/runs/:runId`。
- 新增测试覆盖 combobox 值归一化、合成记录筛选和音频信息格式化。
- 运行 `pnpm typecheck`、`pnpm test`、`pnpm --filter @tts-platform/web build`，均通过。

### 缺陷和风险点

- 合成成功后当前仍会跳转到运行详情页，因此用户回到合成页后才能看到底部记录刷新后的列表。
- 底部表格只展示同步合成记录，音色克隆记录仍通过详情链接或 archive 查看。
- Web build 仍有 Vuetify/MDI chunk 大小提示，不影响功能。

### 下一步内容或计划内容

- 如果希望运行后停留在合成页，可把提交成功后的跳转改成刷新底部记录并高亮最新 run。
- 后续可以抽出通用 RunTable 组件，避免合成页和历史 RunsPage 的表格逻辑重复。

### 推荐内容

- 对 Vuetify combobox/select 一类控件，提交前应始终做值归一化，不直接假设 v-model 是字符串。
- 合成页的记录区聚焦 `tts.sync`，其他 operation 继续通过 archive/detail 做审计查看。
