## MiniMax 流式合成无音频修复

### 提问和更改方向

用户反馈 MiniMax 流式合成返回 `session.completed`，提示成功但没有音频。用户给出的 MiniMax 请求启用了 `stream: true` 和 `output_format: "hex"`，响应只有 completed 事件，说明平台下游完成了生命周期但没有收到或归档任何 `audio.chunk`。

### 当前项目状况和进展

- 后端 `/v1/tts/stream/:sessionId/ws` 已经支持把 adapter 产出的 `audio.chunk` 转发给前端，并在流式 run archive 中写入音频文件。
- MiniMax adapter 此前的 `synthesizeStream()` 只产出 `session.started`、`metadata` 和 `session.completed`，没有连接 MiniMax WebSocket，也没有解析 `data.audio`。
- MiniMax 官方 WebSocket 示例协议为：连接成功后收到 `connected_success`，发送 `task_start`，收到 `task_started` 后发送 `task_continue` 文本，从后续帧的 `data.audio` 读取 hex 音频，遇到 `is_final` 后发送 `task_finish` 并完成。

### 当前步骤所做内容

- MiniMax adapter 新增可注入的 WebSocket factory，默认使用 `ws` 连接 `${baseUrl}/ws/v1/t2a_v2`。
- `synthesizeStream()` 改为真实执行 MiniMax WebSocket 协议：处理连接、上游 metadata、错误帧、`task_start`、`task_continue`、`task_finish` 和完成事件。
- 新增 `streamStartTask()`、`streamContinueTask()`、`streamFinishTask()`、`audioHexFromStreamPayload()`、`parseJsonFrame()`、`AsyncEventQueue` 等 helper，并按项目要求补中文注释。
- MiniMax 流式上游帧中的 `data.audio` 会按 hex 解码为 `Uint8Array`，以 `audio.chunk` 事件输出，后端 route 会继续负责转发和归档。
- 更新 MiniMax adapter 单元测试，用 fake WebSocket 模拟 `connected_success`、`task_started`、hex audio 和 `is_final`，断言能产生 `audio.chunk` 且字节内容正确。

### 缺陷和风险点

- 当前按 MiniMax 官方示例处理 `data.audio` 和 `is_final`；如果 MiniMax 后续增加二进制音频帧或不同完成事件，需要扩展解析分支。
- 当前前端仍通过后端流式 route 接收二进制 chunk；若浏览器端要实时播放 MiniMax MP3 chunk，后续可能需要更细的 MediaSource/播放器策略。
- `task_start` 从已审计的 vendor request 派生，会保留 `language_boost`、`subtitle_enable`、`output_format` 等 vendor extension；若 MiniMax WS 对某些 HTTP 字段不兼容，需要再按 WS Contract 精简字段。

### 下一步内容或计划内容

- 用真实 MiniMax API key 跑一次端到端流式合成，确认归档目录中出现 `audio.mp3` 且浏览器可播放。
- 可以把 MiniMax Contract 中“真正面向前端的 WebSocket pipe/proxy 仍需后续补齐”的旧描述更新掉，避免文档落后于实现。

### 推荐内容

- MiniMax 流式合成继续使用 `output_format: "hex"`，并检查运行详情中的 `vendorEvents` 是否包含上游 `data.audio` 帧和平台 `audio.chunk`。
- 如果后续出现 completed 但无音频，优先检查上游帧是否真的包含 `data.audio`，再检查 route 是否收到 `audio.chunk`。

## 豆包 SSE 空 Resource Id 与错误信息修复

### 提问和更改方向

用户反馈豆包合成时提示 `Doubao SSE event returned an error code.`。排查 stream archive 发现豆包 metadata 中 `resourceId` 为空字符串，说明 vendor extension 中的空 `resourceId` 覆盖了默认 `seed-icl-2.0`。

### 当前项目状况和进展

- 豆包 adapter 已能通过 capability 暴露默认 Resource Id。
- 真实失败路径显示平台错误信息过于笼统，没有把豆包 SSE payload 中的 `code/message` 暴露给调用方。

### 当前步骤所做内容

- 修改 `apps/api/src/adapters/doubao/adapter.ts`：`uid`、`namespace`、`resourceId`、`ttsModel`、`speakerId`、`customSpeakerId` 等字符串扩展必须非空才会应用。
- 空字符串扩展现在进入 `mappingReport.ignoredFields`，不会覆盖默认 request。
- 豆包 SSE 错误现在会抛出 `Doubao SSE event returned error code {code}: {message}`，便于定位 speaker/resource/auth/quota 问题。
- 新增 `doubao-adapter.test.ts` 回归测试，覆盖空扩展不覆盖默认值、错误码和错误消息透出。
- 运行 `pnpm --filter @tts-platform/api test` 和 `pnpm --filter @tts-platform/api typecheck`，均通过。

### 缺陷和风险点

- 已失败的历史 run archive 仍会保留当时的空 `resourceId`，不会被自动修复。
- 如果前端高级参数编辑器仍提交空字符串，后端会忽略这些空值；UI 后续可考虑不提交空 vendor extension 字段。

### 下一步内容或计划内容

- 重启或等待 API dev server 热更新后重新发起豆包合成。
- 如果仍失败，根据新的错误消息中的豆包 `code/message` 判断是 speaker 权限、Resource Id 不匹配、鉴权还是配额问题。

### 推荐内容

- 豆包合成不确定时先不要填写 `vendor.extensions.doubao.params.resourceId`，让 adapter 使用默认 `seed-icl-2.0`。
- 只在确认具体 Resource Id 与 speaker 匹配时，再通过 vendor extension 覆盖。
