## 增加 .env 支持并排除密钥文件

### 提问和更改方向

用户要求支持 `.env` 配置 MiniMax API Key，并确保 env 文件不进入 git。

### 当前项目状况和进展

- `.gitignore` 已排除 `.env` 和 `.env.*`，保留 `.env.example` 可提交。
- API 启动时会加载本地 env 文件，然后实例化 MiniMax adapter。

### 当前步骤所做内容

- 新增 `apps/api/src/config/env.ts`，按顺序读取 `.env`、`.env.local`、`apps/api/.env`、`apps/api/.env.local`。
- 新增 `.env.example`，包含 `MINIMAX_API_KEY`、`PORT`、`HOST` 示例。
- `buildApp()` 增加 `loadEnv` 选项，默认加载 env，测试可关闭。
- 新增 `apps/api/src/test/env.test.ts`，验证 env 文件加载、不覆盖已有环境变量、引号转义解析。
- 更新 MiniMax `Contract.md` 和 `sync-curl.md`，说明 `.env` 使用方式。

### 缺陷和风险点

- 当前 env loader 只实现轻量 dotenv 语法，不支持复杂 shell 展开或变量引用。
- 真实密钥仍只应保留在本地 env 或部署环境变量中，不应进入 archive、request body 或前端。

### 下一步内容或计划内容

- 后续如果需要更多配置项，可统一放入 API config 模块并继续从 env 读取。
- 若引入部署环境，继续保持密钥由进程环境注入，不落库、不归档。

### 推荐内容

- 本地使用 `cp .env.example .env` 后填写 `MINIMAX_API_KEY`。
- 不要在 tests、examples 或 Contract 文档中写入真实 key。
