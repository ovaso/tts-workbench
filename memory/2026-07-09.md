## Doubao 合成 Resource 语义纠偏

### 提问和更改方向

用户指出上一节继续让豆包合成走 `seed-icl-*` 是错误方向。ICL 模型只用于音色复刻创建；音色 ID 已经存在后，TTS 合成应该使用 `seed-tts-*` 合成模型和既有 speaker id，不应该把合成请求的 `X-Api-Resource-Id` 切换到 ICL。

### 当前项目状况和进展

- 豆包 capability 需要表达为 provider 级音色兼容，而不是 same_resource。
- `seed-icl-*` 只支持 `voice.clone.create`，不能进入普通 TTS 合成资源。
- TTS 合成中的 `model` 映射到 `X-Api-Resource-Id`，只允许 `seed-tts-*`。
- 历史 voice 上遗留的 `compatibility.scope=resource` / `resourceIds=["seed-icl-*"]` 只能作为旧数据兼容信息，不能影响新的合成 plan。

### 当前步骤所做内容

- 将 Doubao capability 的 `voiceCompatibilityPolicy` 从 `same_resource` 改为 `provider_wide`。
- 将 Doubao clone result 的 `VoiceRecord.compatibility` 改为 `scope=provider`，保留 `createdWithModelId=seed-icl-*` 用于审计，保留 `preferredModelId=seed-tts-2.0` 用于默认合成模型建议。
- Doubao TTS plan 不再读取 `voice.compatibility.resourceIds` 来覆盖 `X-Api-Resource-Id`。
- Doubao TTS vendor extension 的 `resourceId` 只接受 `seed-tts-*`；传入 `seed-icl-*` 会进入 `mappingReport.ignoredFields`。
- 历史展示音色 ID 如 `doubao_seed-icl-2.0_S_xxx` 仍会剥离前缀，发给厂商的 speaker 保持为真实 `S_xxx`。
- `/v1/voices` 输出时允许 adapter 推导的兼容事实覆盖历史 registry 里的旧 compatibility，因此旧 Doubao `scope=resource` 会在展示和前端选项层降级为 provider 级。
- 更新 Doubao adapter 测试，覆盖历史 ICL compatibility 和显式 ICL resource override 都不会让合成走 ICL。
- 更新 API 测试，覆盖历史 Doubao resource-bound voice 会被规范化为 provider-wide compatibility。
- 更新 Doubao Contract，明确 `seed-icl-*` 只用于复刻创建，合成 resource 必须是 `seed-tts-*`。

### 缺陷和风险点

- 旧 run archive 中已经记录的失败请求仍显示 `resourceId=seed-icl-2.0`，这是历史错误实现产生的归档，不代表当前行为。
- 如果豆包仍拒绝 `seed-tts-* + speaker`，下一步应排查 speaker 本身是否属于当前账号、是否训练成功、是否被删除，而不是回退到 ICL 合成 resource。

### 下一步内容或计划内容

- 用当前修正后的 adapter 重新发起豆包合成，确认 vendor request 中 `X-Api-Resource-Id=seed-tts-2.0`。
- 如仍失败，再接入或临时调用 `/api/v3/tts/get_voice` 查询 speaker 状态和账号归属。

### 推荐内容

- UI 文案中应把 `seed-tts-*` 标注为“合成资源”，把 `seed-icl-*` 标注为“复刻资源”，避免把 clone model 和 TTS resource 混为一谈。
- 后续迁移历史 voice registry 时，应该把 Doubao 旧 `resource` compatibility 降级为 provider 级 compatibility。

### 下一步内容或计划内容

- 增加 job 级指标采集：耗时、音频时长、RTF、大小、失败 code。
- 运行页后续可补 job 详情展开，直接显示 request、mapping report 和音频，而不是只跳转普通 run。

### 推荐内容

- 短期继续保持顺序执行，先验证 mock 和少量真实厂商闭环，再决定是否引入队列。
- 后续如果需要公平比较，应优先使用 `canonical_only` 配置集合，并在计划创建时显示每个 config 的 vendor mode。

## Benches 指标采集补强

### 提问和更改方向

用户指出 Benches 运行结果缺少首包延迟等核心指标，Benchmark 的主要目的应是看指标，而不只是看成功失败和 runId。

### 当前项目状况和进展

- 上一步已经有 `BenchmarkPlan`、顺序 runner 和普通 run archive 引用，但 job 结果只记录成功/失败、runId、result 和错误。
- 同步合成 `TTSFacade.synthesizeSync()` 只能在完整响应完成后返回，无法观测真实首个音频包。
- 真实首包延迟需要走 `tts.stream` 的统一流式事件，并在第一个 `audio.chunk` 到达时记录时间。

### 当前步骤所做内容

- core 新增 `BenchmarkPlanJobMetrics`，记录 `textLength`、`totalLatencyMs`、`firstPacketLatencyMs`、`firstPacketSource`、`audioDurationMs`、`audioDurationSource`、`audioByteLength`、`audioChunkCount`、`realtimeFactor`。
- core 扩展 `BenchmarkPlanSummary`，记录 `measuredJobs`、平均总耗时、平均首包、平均音频时长、平均 RTF 和总音频字节数。
- API `TTSFacade` 新增 `synthesizeStreamToArchive()`，直接消费 adapter 的 `synthesizeStream()` 事件，测量首个 `audio.chunk` 延迟，并写入普通 `tts.stream` run archive。
- Benchmark runner 支持 `tts.stream` plan；同步 job 记录总耗时、音频字节、音频时长和 RTF，但用 `sync_not_observable` 明确标记首包不可观测。
- 音频时长优先来自 stream completed event，其次解析 WAV 文件头，再其次读取 `vendor-response.json` 中的 `durationMs` 风格字段。
- Benches 创建方案弹窗新增“同步合成/流式合成”选择。
- 运行列表新增平均指标列；job 明细新增首包、总耗时、音频时长、RTF、字节数列。
- 补充 API 集成测试，覆盖 sync 指标和 stream 首包指标；补充前端 helper 测试覆盖指标格式化。

### 缺陷和风险点

- 同步 HTTP 合成无法真实采集首包延迟，当前不会伪造该指标，只标记为同步不可观测。
- 压缩音频如 mp3/ogg 的时长如果厂商响应没有 durationMs，当前不会调用 ffprobe，因此可能缺少音频时长和 RTF。
- 当前仍是顺序 runner，没有并发、预热、重复次数、分位数、失败分类和稳定性统计。
- 多次运行同一个 plan 仍会覆盖 plan 内 job 指标；普通 run archive 会保留每次成功或 stream 失败归档。

### 下一步内容或计划内容

- 增加重复次数和聚合统计，如 p50/p95/p99、min/max、成功率、错误码分布。
- 增加 warmup 参数，避免冷启动污染主要指标。
- 对非 WAV 音频可引入可选 ffprobe 指标抽取，但应保持 archive 事实来源和测试稳定。
- 后续引入独立 `BenchmarkRun` 实体，避免多次运行覆盖同一 plan 的结果。

### 推荐内容

- 需要首包延迟时优先创建 `tts.stream` Benchmark 方案，不要用同步合成结果替代首包。
- 指标展示应继续保留来源字段，避免把同步响应耗时误读为首包延迟。
